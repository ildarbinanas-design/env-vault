# ADR 0006: Versioned operational release contract and closed historical routing

- Status: accepted
- Date: 2026-07-18
- Scope: release parameters, workflow identities, historical source contracts,
  typed runtime projections, and contract evolution

## Context

The original `release/contract.v1.json` accumulated two different concerns:
current release parameters and immutable incident/history policy. Runtime shell
and workflow code still repeated repository, branch, workflow, target, asset,
formula, and App values. A historical release must be checked with the contract
and checker that existed at its source, but allowing any structurally valid v1
document to authorize current behavior would make a downgrade path.

`v0.0.14` and `v0.0.15` used contract generation v1 and evidence format v1.
`v0.0.16` still used contract generation v1, but published the compact evidence
format v2. Contract generation and evidence format therefore cannot be inferred
from one another or from the current branch.

## Decision

### Current operational authority

`release/contract.v2.json` is the canonical contract for new control-plane
code. It owns version/tag policy, source and tap repositories/default branches,
naming, the five targets and ten assets, Homebrew templates, twelve exact
workflow identities, two App identities, required checks, and concurrency.

`releasecheck contract operational` emits
`env-vault.release-contract-operational.v2`. A runtime typed pair consists of
that projection and `releasecheck --version --json`. Both documents bind the
same semantic contract digest and the same SHA-256 of the exact v2 file bytes.
Mutation scripts call `release_require_typed_contract_projection` before
GitHub access. Local operator commands use `with-typed-contract.sh`, which
builds one private checker, replaces caller-supplied projection variables, and
removes the temporary pair on success, failure, or signal.

The shell trust boundary requires a dedicated `RELEASE_CONTRACT_CHECKER`
regular executable built from the same clean v2 checkout as its consumer. That
checker strictly validates the exact local `release/contract.v2.json`, then
freshly emits its complete checker-version and operational-projection
documents. The shell compares the supplied documents byte-for-byte with those
fresh outputs before loading the projection. This rejects whitespace changes,
multiple roots, missing, unknown, duplicate, or case-variant members, and any
repository, workflow, target, or asset divergence before consumer or GitHub
access. Self-declared matching semantic or raw-file digests are never
sufficient authority for different bytes or operational content. The jq-only
single-root projection remains available solely to read-only/offline helpers;
it is explicitly non-authoritative and cannot pass
`release_require_typed_contract_projection` for a mutation.

Top-level Actions triggers, job environments, action subject lists, and the
shared concurrency group cannot be interpolated from runtime shell. Tests bind
those static YAML fields to the v2 contract. The contract inventory is exactly
twelve workflows; exactly planning, publisher, release-asset bootstrap,
Homebrew bridge, and release evidence participate in the non-cancelling shared
release queue. The two provenance/SBOM action invocations are bound to the
five contract archives.

### Historical authority

`release/history/contract.v1.json` is the immutable v1 archive. Its raw digest
is `69be8d6ccf4480cbdbff3c722d8080d62b31051cef0101fadb47a30cdc0e2715`
and its semantic digest is
`6b83efee82bf8a0d9c1fcc3f491f313dee3dd29f31f0837b27051c7c65e61ef5`.
`release/contract-history.v2.json` binds those bytes and admits only the exact
closed repository/version/source tuples for `v0.0.14`, `v0.0.15`, and
`v0.0.16`. Each entry records `contract_generation` and `evidence_format`
separately, plus immutable evidence coordinates.

The current checker authorizes the registry and exact archived bytes, then the
checker built from the immutable release source validates its own contract and
historical data. An arbitrary in-memory v1 contract has no compatibility
authority. Historical evidence publication preserves separate identities for
the frozen source/payload checker, the reviewed current control checker, and
the selected evidence publisher checker. The reviewed v2 typed pair is
regenerated by its dedicated checker; these identities are not silently
substituted for one another.

The live `release/contract.v1.json` is retained only as an incident-bearing
transition artifact for source revisions that still contain it. It is neither
the immutable archive nor the canonical contract for new runtime code. The
permanent `v0.0.12`/PR #31 title, author, base, and head predicates remain
explicit historical constants; they must not be derived from current naming or
App policy.

### Evolution semantics

The v2 `evolution.previous_semantic_sha256` points to the immutable archived
release lineage (`6b83...`), not to the mutable live-v1 control snapshot that
immediately preceded this migration. This is intentional and nonstandard: the
field proves continuity with published historical releases, while raw-byte and
tuple authority remains in the separate history registry. A future evolution
must preserve that distinction instead of treating the field as a generic
"previous file in Git" pointer.

## Verification

- Strict decoders reject missing, unknown, duplicate, reordered-identity, and
  unsupported contract/history fields.
- Contract workflow IDs, events, and jobs are exact sets, not subsets.
- Static trigger, environment, App-token, concurrency, required-check, and
  attestation-subject parity is tested against v2.
- Frozen v1 checker-capability fixtures cover `v0.0.15` and `v0.0.16` without
  introducing current-only fields.
- The compact `v0.0.16` fixture reconstructs and verifies fully offline with
  no GitHub credential or network dependency.
- Product implementation paths are unchanged by this migration.

## Consequences

Runtime literals and permissive v1 fallback paths are reduced, while the
release-only Go, workflow, and adversarial-test surface grows. This is an
accepted security trade-off, not a job-count or wall-time improvement claim.
Exact hosted-run metrics can be stated only after a successful release on the
new graph. Published tags, assets, evidence commits, and the v1 archive are not
rewritten.
